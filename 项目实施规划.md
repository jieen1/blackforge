> 不是 fork nano-vLLM，也不是继续魔改完整 vLLM，而是新建一个只服务 `Qwen3.6-27B-NVFP4 + SM120 + 单卡 + 并发≤4` 的专用推理运行时。

参考关系应当是：

- vLLM：模型实现、NVFP4 loader、GDN/MTP 正确性、服务行为和性能基准；
- nano-vLLM：小型 scheduler、block manager、CUDA Graph runner 的代码组织方式；
- FlashInfer/当前项目：attention 与 paged-KV 参考；
- `kekzl/imp`：SM120 低精度 kernel 和整机专用化参考；
- 新项目：只保留你的工作负载需要的功能。

nano-vLLM 官方目前只有普通 Qwen3 模型，定位是约 1,200 行的轻量实现，[官方仓库](https://github.com/GeeeekExplorer/nano-vllm)。而 vLLM 已经有 hybrid model、NVFP4、MTP、CUDA Graph、prefix caching 等基础设施，[vLLM 官方仓库](https://github.com/vllm-project/vllm)。所以应当“参考两者”，而不是把任意一个直接当成最终底座。

# 一、最终目标合同

项目可命名为 `qwen-sm120-runtime`；如果希望保留熟悉感，也可以叫 `qwen-sm120-vllm`，但它本质上不是通用 vLLM。

第一版只支持：

| 项目 | 固定范围 |
|---|---|
| GPU | 单张 RTX PRO 6000 Blackwell，SM120 |
| 模型 | `unsloth/Qwen3.6-27B-NVFP4` |
| 并发 | 1～4 |
| 权重 | NVFP4 |
| KV cache | 首先 FP8 e4m3 |
| 模型结构 | 16 full-attention + 48 GDN |
| 推测解码 | 模型自带 MTP，K=3 |
| 上下文 | 4K、16K、32K 三个优化 bucket |
| 接口 | OpenAI-compatible streaming，满足 coding agent |
| 采样 | greedy、temperature、top-p/top-k |
| 不支持 | 多卡、LoRA、beam search、其他模型、动态量化、复杂多租户 |

这个范围必须冻结。每加入一个“通用功能”，都会削弱专用化收益。

核心设计原则是建立统一 `OpRegistry`：

```python
ops.attention(...)
ops.gdn(...)
ops.nvfp4_linear(...)
ops.rmsnorm_residual(...)
ops.mtp_verify(...)
```

每个算子最开始可以调用 vLLM/FlashInfer/torch，之后逐个替换成自研 kernel。这样不会陷入“所有东西都写完以后才第一次运行模型”的危险模式。

# 三、分阶段实施规划

## Phase 0：冻结基准和决策数据

目标：知道整栈优化到底能从哪里拿到速度。

任务：

1. 固定三套 benchmark：
   - W1：4K input + 1K output，c1/c4；
   - W2：32K input + 1K output，c1/c4；
   - Agent Replay：采集真实多-agent请求的长度、到达时间和输出长度。
2. 固定两个 baseline：
   - 原生 vLLM + FlashInfer；
   - 当前 custom attention backend。
3. 用 `nsys` 分解一个完整 decode/MTP step：
   - CPU/launch gap；
   - NVFP4 GEMM；
   - GDN；
   - attention prefill/decode；
   - MTP；
   - sampling/logits；
   - metadata/cache update。
4. 记录 GPU 时钟、功率和温度，所有 A/B 交错运行。

Gate：

- 如果 CPU/launch gap <3%，不优先重写 C++ scheduler；
- 如果 GDN/GEMM 占比最大，整栈路线必须优先它们；
- 所有后续收益必须按“accepted output tokens/s”和 ITL 衡量，不能只看 draft/s。

交付：

```text
notes/phase-0-baseline.md
benchmarks/baseline/*.json
profiles/vllm-primary.nsys-rep
```

## Phase 1：建立 vLLM Oracle 和逐层验证框架

目标：任何新实现都能和当前正确运行的 vLLM 对比。

任务：

1. 给 `/home/bot/vllm` 增加只读 hook：
   - embedding 输出；
   - 每层输入/输出；
   - GDN state；
   - attention 输出；
   - final norm；
   - logits；
   - MTP logits/acceptance。
2. 保存一组小型 golden fixtures：
   - 16、128、4096 token；
   - batch 1/4；
   - prefill；
   - 连续 decode 256～1000 step；
   - slot reset/reuse。
3. 编写 layer-by-layer comparator。
4. 特别验证 GDN state：
   - 跨 token 累积；
   - request reset；
   - 四个 slot 隔离；
   - CUDA Graph replay 后不串状态。

Gate：

- 复用相同算子时要求接近 bitwise；
- 自研量化算子采用 cosine、误差分位数和 top-k logits 一起判断；
- greedy 固定测试集不能发生系统性输出漂移；
- MTP acceptance 不得比 vLLM 下降超过 1 个百分点。

这一阶段不要追求速度，只建设不会被优化破坏的正确性护栏。

## Phase 2：模型加载和离线权重打包

目标：新 runtime 独立加载 Unsloth NVFP4 checkpoint。

任务：

1. 读取 HF config、safetensors 和 quantization metadata。
2. 移植最小范围的 ModelOpt NVFP4 scale/weight解释逻辑。
3. 正确识别：
   - 16 个 attention layer；
   - 48 个 GDN layer；
   - MTP 权重；
   - 哪些 MTP 权重仍为 BF16。
4. 建立离线 packer，将 checkpoint 转成 SM120-native 文件：
   - NVFP4 MMA 所需 swizzle；
   - fused QKV；
   - fused gate/up；
   - 对齐与 padding；
   - per-layer metadata；
   - checksum/version。
5. 第一次转换慢可以接受，之后直接 mmap/load 最终布局。

Gate：

- 每个 tensor 名称、shape、scale 与 vLLM loader 对齐；
- 加载后逐层权重解码误差通过；
- 不允许 runtime 临时做大规模 weight transpose。

额外显存在这里第一次真正用于加速：可以为不同 GEMM shape 保存两套预打包权重布局，但先不默认开启。

## Phase 3：最小 eager 模型执行器

目标：不依赖 vLLM scheduler，独立跑通完整 forward。

初始算子可以复用：

- vLLM NVFP4 linear；
- FlashInfer/native attention；
- vLLM GDN kernel；
- torch RMSNorm/embedding/sampling。

任务：

1. 实现 Qwen3.6 layer graph。
2. 实现 hybrid cache：
   - 16 层 paged KV；
   - 48 层 GDN recurrent/conv state；
   - 每 request 独立。
3. 先跑 prefill；
4. 再跑单 token decode；
5. 最后跑 256 token 连续生成。
6. 支持 batch 1 和 batch 4，但暂不支持动态请求。

Gate：

- 完整模型 logits、token 输出、GDN state 对齐；
- 4 个请求并行不串 cache；
- request 结束后 slot reset 正确；
- eager 速度只要求不比 vLLM eager 慢超过约 10%。

这是第一个真正的整机里程碑。

## Phase 4：四槽固定调度器和 CUDA Graph

目标：建立符合多-agent场景的极简热路径。

设计：

- 固定四个物理 request slots；
- KV/GDN state 地址终身固定；
- slot 内 request 可以动态进入/退出；
- 为 batch 1/2/3/4 分别 capture graph；
- 每轮只更新固定 metadata buffer；
- 不移动 KV/GDN state，只更新逻辑 slot 映射。

建议 graph 类型：

```text
prefill-4K
prefill-16K
prefill-32K
decode-qo1-bs1/2/3/4
mtp-verify-qo4-bs1/2/3/4
```

任务：

1. persistent input/output/workspace；
2. graph-safe page table；
3. graph-safe sampling；
4. 异步 streaming；
5. request admission 和 cancellation；
6. Python server 与 GPU hot loop 分离。

Gate：

- CUDA Graph replay 正确；
- graph replay 中不得出现分配；
- CPU launch gap显著降低；
- 如果相同 kernel 下仍不比 vLLM快，必须重新 profile，不能继续凭感觉重写 scheduler。

## Phase 5：完成 SM120 packed-M attention v2

这与当前项目直接衔接，是第一套核心自研 kernel。

目标形状：

```text
head_dim = 256
QH:KVH = 24:4
GQA group = 6
qo_len = 4
KV = FP8
page size = 固定生产值
```

实施顺序：

1. continuous KV、固定 32K；
2. `grid.y = KVH = 4`；
3. packed row 映射：
   `row -> token/head_in_group`；
4. online softmax；
5. causal bottom-right mask；
6. paged KV；
7. split-KV；
8. graph-safe workspace；
9. prefill 和 qo_len=1 专用变体。

Gate：

- 正确性、causal probe、page-boundary、sanitizer 全过；
- kernel 至少达到 native parity；
- 项目级目标应是比 native attention subsystem 快 ≥15%；
- 必须端到端复测，不能只留下 microbenchmark。

NVFP4-KV 此时仍然不是性能主线。

## Phase 6：GDN 全栈优化

这是整栈项目最重要的阶段，因为有 48 个 GDN layer。

步骤：

1. profile 每个 GDN layer 的完整 kernel 序列；
2. 建立单层 GDN replay benchmark；
3. 分别测试：
   - input projection 融合；
   - gate/activation 融合；
   - conv/state update 融合；
   - norm + residual 融合；
   - 固定 state layout；
   - 去除临时 tensor；
   - persistent workspace。
4. 最终考虑把一个完整 GDN block 压缩成少数几个 kernel。

Gate：

- 1000-step state 演化正确；
- 四 slot reset/复用正确；
- 单个 GDN layer 至少提升 10% 才进入 runtime；
- 每次融合都做完整模型质量回归。

如果 profiling 证明 GDN 是主要时间占比，这一阶段的优先级高于 NVFP4 attention。

## Phase 7：NVFP4 GEMM 与模型级融合

不要一开始重写所有 GEMM。先按占比排序：

1. GDN/attention input projection；
2. MLP gate-up；
3. MLP down projection；
4. attention output projection；
5. MTP projection；
6. lm_head。

每个 shape 建立独立 autotune 表：

```text
prefill M
decode M=1..4
MTP verify M=4..16
```

额外显存可用于：

- decode 和 prefill 两套权重打包；
- fused QKV / fused gate-up；
- 不同 tile 所需的独立 swizzle；
- graph pool 和持久 workspace；
- 高频小矩阵的特殊副本。

Gate：

- 只有端到端收益 ≥1% 的权重副本才保留；
- 计入额外权重加载时间和显存；
- 不因为“用了更多显存”就判定成功。

## Phase 8：MTP 与单请求加速

目标：提高四个 agent 各自的 accepted token 生成速度。

顺序：

1. 先完整复现 vLLM 的 MTP K=3；
2. 固定 draft/verify graph；
3. 融合：
   - draft logits；
   - verify attention；
   - acceptance；
   - token compact/scatter；
4. 再评估 DFlash，不能同时重写 MTP 和 DFlash。

关键指标：

```text
accepted tokens / second
acceptance rate
ms / accepted token
单请求 ITL
四请求 aggregate throughput
```

Gate：

- speculative 解码关闭时必须有稳定 fallback；
- acceptance 不能因量化或 kernel 改动明显下降；
- 如果 decode kernel 变快但 acceptance 下降，按最终 accepted token 计算净收益。

## Phase 9：利用剩余显存换速度

到这里才系统测试“45GB剩余显存怎么用”。

按优先级：

1. 多套 kernel-native 权重布局；
2. 更大的 prefix cache；
3. coding agent 公共系统提示/repo context缓存；
4. CUDA Graph 多 bucket；
5. persistent workspace 和 metadata；
6. KV hot-window 连续布局；
7. decode/prefill 双 KV 布局；
8. 部分热点权重的 BF16/FP8 副本。

每项都必须受控 A/B：

```text
额外显存
额外写入成本
TTFT
ITL
accepted tok/s
最终端到端提升
```

特别注意：双 KV layout 每生成一个 token 都可能多一次写入；只有读取节省超过双写成本才成立。

## Phase 10：真实 agent 优化

针对你的使用场景，而不是公开 serving benchmark：

1. 记录四个 agent 的真实请求轨迹；
2. 分析共享前缀：
   - system prompt；
   -工具定义；
   - repo map；
   -编程规范；
3. block-level prefix cache；
4. agent slot affinity；
5. request 暂停/恢复保留 cache；
6. 大上下文热区和冷区分层；
7. 长 prefill 不阻塞其他 agent decode。

最终验收必须基于 replay，而不是随机 token benchmark。

## Phase 11：服务化和稳定性

任务：

- OpenAI-compatible endpoint；
- streaming；
- cancellation；
- timeout；
- context overflow；
- OOM recovery；
- health check；
- 自动回退到 vLLM；
- kernel/runtime版本与 packed-weight 版本校验；
- 崩溃后 state 清理；
- 长时间稳定性测试。

推荐保留 vLLM 服务作为 fallback：

```text
qwen-sm120-runtime 健康 → 走专用引擎
初始化/正确性/显存异常 → 回退本机 vLLM
```

# 四、性能验收标准

建议正式设三档目标。

| 阶段 | 验收目标 |
|---|---|
| MVP | 正确运行，性能达到 native vLLM ±5% |
| v1 | Agent Replay c4 的 p50 ITL 提升 ≥10%，质量不回退 |
| v2 | W2/Agent Replay 提升 ≥20%，TTFT 不明显恶化 |
| Stretch | 通过 GDN、MTP、权重布局和prefix cache达到 25～35% 场景收益 |

不能承诺一定达到 30%，因为目前还没有 GDN/GEMM/launch gap 的精确占比。但整栈路线的理论上限显著高于 attention-only。

其中当前 `sm120-flash-attention` 不废弃：它继续作为 kernel 实验室。只有通过 correctness/performance gate 的 kernel 才进入 `qwen-sm120-runtime`。这样能把“kernel 研究”和“可稳定运行的推理引擎”隔离开。

这个项目的本质不是重新实现通用 vLLM，而是构建一台软件意义上的“Qwen3.6-27B-NVFP4 专用推理机”。在你的单模型、单GPU、最多四并发场景下，这条路线确实比继续无限扩展一个 vLLM attention backend 更有性能上限。
